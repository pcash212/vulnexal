Vulnerability Information

Learn about common web vulnerabilities and how to protect against them

What is HTTPS?

HTTPS (HyperText Transfer Protocol Secure) is an encrypted version of HTTP. It uses SSL/TLS protocols to encrypt data transmitted between a user's browser and the web server.

Why is it Important?
  • Data Encryption: Prevents eavesdropping and man-in-the-middle attacks
  • Data Integrity: Ensures data hasn't been tampered with during transmission
  • Authentication: Verifies that users are communicating with the intended website
  • SEO Benefits: Google prioritizes HTTPS sites in search rankings
  • User Trust: Browsers display security warnings for non-HTTPS sites
Risks of Not Using HTTPS:
  • Login credentials transmitted in plain text
  • Session cookies vulnerable to theft
  • Personal information exposed to attackers
  • Payment details can be intercepted
  • Browser warnings deter users from visiting your site
How to Fix:
  1. Purchase an SSL/TLS certificate from a trusted Certificate Authority (CA)
  2. Install the certificate on your web server
  3. Configure your server to use HTTPS
  4. Redirect all HTTP traffic to HTTPS
  5. Enable HSTS (HTTP Strict Transport Security) header

What is Clickjacking?

Clickjacking (also known as UI redressing) is an attack where a malicious website tricks users into clicking on something different from what they perceive. The attacker loads your website in an invisible iframe and overlays it with deceptive content.

How Does it Work?

An attacker creates a malicious page that loads your website in a transparent iframe. They then position visible buttons or links over your site's interactive elements. When users think they're clicking on the attacker's content, they're actually interacting with your hidden website.

Potential Consequences:
  • Unauthorized actions performed on behalf of users
  • Unwanted social media posts or follows
  • Malware downloads
  • Unauthorized purchases or money transfers
  • Changing account settings or permissions
How to Protect:
1. X-Frame-Options Header: 
X-Frame-Options: DENY
X-Frame-Options: SAMEORIGIN
2. Content Security Policy (CSP): 
Content-Security-Policy: frame-ancestors 'none';
Content-Security-Policy: frame-ancestors 'self';
3. Frame-busting JavaScript: 
if (top !== self) {
  top.location = self.location;
}

What is XSS?

Cross-Site Scripting (XSS) is a security vulnerability that allows attackers to inject malicious scripts into web pages viewed by other users. These scripts execute in the victim's browser with the same privileges as the legitimate website.

Types of XSS:
  • Stored XSS: Malicious script is permanently stored on the target server (e.g., in a database)
  • Reflected XSS: Script is reflected off the web server (e.g., in URL parameters or form submissions)
  • DOM-based XSS: Vulnerability exists in client-side code rather than server-side
Attack Scenarios:
  • Stealing session cookies and authentication tokens
  • Redirecting users to phishing websites
  • Capturing keystrokes and form data
  • Defacing websites
  • Distributing malware
  • Performing actions on behalf of users
Prevention Methods:
  1. Input Validation: Validate and sanitize all user inputs
  2. Output Encoding: Encode data before rendering in HTML
  3. Content Security Policy: Implement CSP headers to restrict script execution
  4. HTTPOnly Cookies: Set HTTPOnly flag on cookies to prevent JavaScript access
  5. Use Security Libraries: Utilize frameworks with built-in XSS protection
  6. Avoid Dangerous Functions: Don't use eval(), innerHTML, or document.write() with user input
Best Practice: Always treat user input as untrusted and potentially malicious!

What is Password Field Security?

Password field security refers to the measures taken to protect password input fields from being intercepted, stolen, or exposed during transmission and storage.

Common Vulnerabilities:
  • Unencrypted Transmission: Passwords sent over HTTP can be intercepted
  • Autocomplete Enabled: Browsers may save passwords in plain text
  • Weak Password Policies: Allowing simple, easily guessable passwords
  • Password Visible in Source: Pre-filled passwords in HTML
  • No Rate Limiting: Allowing unlimited login attempts
Security Risks:
  • Password interception via network sniffing
  • Browser-based password theft
  • Brute force attacks
  • Dictionary attacks
  • Credential stuffing attacks
Best Practices:
  1. Always Use HTTPS: Encrypt password transmission with SSL/TLS
  2. Password Field Attributes: 
    <input type="password" 
       autocomplete="new-password" 
       required />
  3. Implement Password Requirements:
    • Minimum length (at least 8-12 characters)
    • Mix of uppercase, lowercase, numbers, and symbols
    • Check against common password lists
  4. Hash Passwords: Never store passwords in plain text (use bcrypt, Argon2, or PBKDF2)
  5. Rate Limiting: Limit login attempts to prevent brute force
  6. Multi-Factor Authentication: Add an extra layer of security
  7. Password Strength Indicator: Help users create strong passwords
Critical: Never transmit passwords over HTTP or store them in plain text!

What is Server Availability?

Server availability refers to the ability of a web server to respond to requests and serve content to users. High availability is crucial for maintaining user trust and business continuity.

Common Availability Issues:
  • DDoS Attacks: Overwhelming server with traffic
  • Server Crashes: Software bugs or hardware failures
  • Network Issues: DNS problems, routing issues
  • Resource Exhaustion: CPU, memory, or bandwidth limits reached
  • Maintenance Windows: Planned downtime
Impact of Downtime:
  • Lost revenue and business opportunities
  • Damaged reputation and user trust
  • Poor SEO rankings
  • Increased support costs
  • Loss of competitive advantage
Improving Availability:
  1. Load Balancing: Distribute traffic across multiple servers
  2. Redundancy: Use backup servers and failover systems
  3. CDN (Content Delivery Network): Cache content closer to users
  4. DDoS Protection: Use services like Cloudflare or AWS Shield
  5. Monitoring & Alerts: Implement uptime monitoring (e.g., Pingdom, UptimeRobot)
  6. Auto-scaling: Automatically increase resources during traffic spikes
  7. Regular Backups: Maintain recent backups for quick recovery
  8. Health Checks: Automatically detect and replace unhealthy servers
Target: Industry standard is 99.9% uptime (less than 9 hours downtime per year)

What are Suspicious URL Patterns?

Suspicious URL patterns are specific characteristics or structures in URLs that may indicate malicious intent, phishing attempts, or security vulnerabilities. These patterns are often used by attackers to deceive users or exploit security weaknesses.

Common Suspicious Patterns:
  • Embedded Scripts: URLs containing <script>, javascript:, or data: protocols
  • SQL Injection Patterns: UNION SELECT, OR 1=1, DROP TABLE
  • Path Traversal: ../, ..\, attempts to access parent directories
  • Command Injection: Semicolons, pipes, backticks, system commands
  • Encoded Payloads: Excessive URL encoding (%20, %3C, %3E) to hide malicious content
  • Phishing Indicators: Typosquatting, homograph attacks, misleading subdomains
Security Risks:
  • XSS attacks via URL parameters
  • SQL injection through query strings
  • Directory traversal and file access
  • Remote code execution
  • Session hijacking
  • Phishing and social engineering
Detection & Prevention:
  1. URL Validation: Check against known malicious patterns
  2. Input Sanitization: Remove or escape special characters
  3. Whitelist Approach: Only allow known-good URL structures
  4. URL Encoding: Properly encode user-generated URLs
  5. Web Application Firewall (WAF): Filter malicious requests
  6. Regular Expression Filtering: Block dangerous patterns
  7. DNSSEC: Prevent DNS spoofing
  8. User Education: Train users to recognize suspicious URLs
Examples of Suspicious URLs: 
❌ http://example.com/<script>alert('XSS')</script>
❌ http://example.com/user?id=1' OR '1'='1
❌ http://example.com/files/../../../../etc/passwd
❌ http://paypa1.com (typosquatting)
❌ http://example.com?redirect=javascript:void(0)
✅ https://example.com/products/123
✅ https://example.com/user?id=abc123
Pro Tip: Always validate and sanitize URL parameters before processing!
